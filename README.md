# AmoPoint Test

Тестовое задание на вакансию PHP-разработчика.

## Задание 1

Необходимо реализовать Laravel-проект, в котором:

1. Консольная команда каждые 5 минут получает данные из внешнего API и сохраняет их в таблицу БД.
2. Route отдает массив записей таблицы в формате JSON.

API для реализации:

`https://official-joke-api.appspot.com/random_joke`

## Что реализовано

В проекте реализован сбор случайных шуток из внешнего API с сохранением в таблицу `jokes`.

Сохраняются поля:

- `id`
- `external_id`
- `type`
- `setup`
- `punchline`
- `created_at`
- `updated_at`

Логика работы:

- команда `php artisan jokes:fetch-random` получает одну случайную шутку из API и сохраняет её в БД
- Laravel Scheduler запускает эту команду каждые 5 минут
- каждая шутка сохраняется отдельной строкой без проверки на дубли
- `GET /api/jokes` возвращает JSON-массив из 100 записей
- `GET /jokes` показывает сохранённые шутки в HTML-таблице с пагинацией и сортировкой

## Routes

### `GET /api/jokes`

Возвращает массив записей таблицы `jokes` в формате JSON.

Пример:

`http://127.0.0.1:8000/api/jokes`

### `GET /jokes`

Отображает страницу со списком шуток.

Возможности:

- пагинация
- сортировка по `ID`
- сортировка по `External ID`
- сортировка по `Type`
- по умолчанию сверху отображаются старые записи

Пример:

`http://127.0.0.1:8000/jokes`

## Проверка решения

## Алгоритм решения

Решение построено не на жёстко заданной карте полей, а на правиле из условия задачи:

1. Находит `select[name="type_val"]`.
2. Собирает все элементы с атрибутом `name`, кроме самого селекта.
3. Для каждого поля определяет визуальный контейнер `p`, чтобы скрывать строку целиком вместе с подписью.
4. Дополнительно скрывает пустые служебные блоки вида `<p>&nbsp;</p>`, чтобы они не ломали вёрстку.
5. При изменении значения селекта проверяет условие `element.name.includes(selectedType)`.
6. Если условие выполняется, поле показывается; если нет, скрывается и отключается.
7. Инициализация выполняется сразу после подключения скрипта, поэтому корректное состояние выставляется уже при загрузке страницы.

## Почему выбран именно такой подход

- решение соответствует формулировке задания буквально: логика строится по `name`, а не по заранее прописанному списку
- при добавлении новых полей с подходящим `name` код не требует изменений
- скрывается весь визуальный блок, а не только `input`, поэтому интерфейс не “ломается”
- пустые разделители `<p>&nbsp;</p>` убираются автоматически, поэтому итоговая вёрстка выглядит аккуратно
- отключение скрытых элементов дополнительно исключает случайное взаимодействие с ними

## Альтернативы и почему они не выбраны

### Жёсткое сопоставление типа и списка полей

Пример подхода:

```js
const map = {
    1: ['input_1', 'button_12', 'button_1'],
    2: ['input_2', 'button_12', 'button_28'],
};
```

Почему не выбран:

- решение становится привязанным к конкретному набору элементов
- любое изменение HTML требует правки JS
- это хуже масштабируется и хуже соответствует тексту задания

### Решение на jQuery

Почему не выбрано:

- в задаче достаточно стандартного DOM API
- чистый JavaScript показывает, что решение не зависит от сторонних библиотек
- для такого объёма логики jQuery не даёт практического выигрыша
